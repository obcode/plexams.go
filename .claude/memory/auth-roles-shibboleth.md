**UPDATE 2026-07-14 — LIVE deployed auf `plexams.cs.hm.edu` (nginx+oauth2-proxy+auth.enabled:true), dabei die SSR-Identitätsfalle entdeckt+deploy-seitig gelöst.** Symptom 1: `GET /` → 500, `graphql-request`: *"Invalid execution result: result is not object or array"* mit HM-Shibboleth-"Veraltete Anfrage"-HTML. Symptom 2 (nach Umstellung auf internen SSR-Endpoint): `GraphQL Error (Code: 401)` Body 45 B = `unauthenticated: no identity from auth proxy` ([graph/auth.go](graph/auth.go) authMiddleware, fail-closed). **Ursache:** Die GUI rendert server-side (SvelteKit SSR `+page.server.ts load()`, `hooks.server.js`, `/api`-Proxys via `gqlProxy.js`) im `gui`-Container — diese Calls haben KEIN OIDC-Cookie, können oauth2-proxy nie passieren → Redirect zur sso.hm.edu-Loginseite → HTML statt JSON. Zusätzlich (vom GUI-Agent verifiziert): der `gui`→backend-Hop reicht **keinerlei** Identität weiter (`request(env.PLEXAMS_SERVER, doc, vars)`, nur content-type; Identität nur via `me{email name role}` @ `+layout.server.ts`). **Lösung (Zwei-URL-Split, in `deploy/` umgesetzt):** `PUBLIC_PLEXAMS_SERVER=https://<host>/query` (Browser, hat Cookie) vs. `PLEXAMS_SERVER=http://plexams:8080/query` (SSR, intern, umgeht oauth2-proxy). nginx injiziert `X-Remote-User` jetzt AUCH auf `location /` (gui-Pfad, autoritativ aus $email), die GUI muss ihn auf ihren internen Backend-Calls **weiterreichen**. Geänderte Dateien: `deploy/nginx/templates/default.conf.template` (location / bekommt X-Remote-User/-Displayname/-Department), `deploy/docker-compose.yml` (PLEXAMS_SERVER default `${PLEXAMS_SERVER:-http://plexams:8080/query}`), `deploy/.env.example`, `deploy/.plexams.yaml.example`, `deploy/README.md` (neuer Abschnitt "SSR-Identität"). **Sequenzierung:** Bis das GUI-Release den Header forwarded, entweder `auth.enabled:false` (SSO-Wand bleibt an nginx, alle eingeloggten = ADMIN, kein Rollen-Split) ODER beide URLs öffentlich lassen. Sobald GUI-Forwarding live → `auth.enabled:true`, keine weitere Deploy-Änderung nötig. **GUI-Agent-Anweisung offen:** in `hooks.server.js` `X-Remote-User` aus `event.request.headers` lesen → `locals`, und alle server-side `graphql-request`-Calls (SSR loads + gqlProxy) von `request(url,...)` auf `new GraphQLClient(url,{headers:{'X-Remote-User':...}})` umstellen; Client-Bundle (Browser) unverändert.
